# 结构诊断与读者 Context 检查

## 结构诊断流程

### 1. 阅读与分析

- 读取当前段落/章节内容
- 拆解段落，识别主题（心理变化、时间线、文化冲击等）
- 标记问题：主题混乱、节奏不佳、读者 Context 缺失

### 2. 提出重组方案

给出 2-3 个结构调整方案，每个方案说明：

- 核心线索（如「从紧张到放松」「按时间线」等）
- 需要删除/移动的内容及理由

**不要直接动手修改，先讨论确认。**

### 3. 素材保留

删除的内容不丢弃，移到「XXX-素材.md」文件中，注明：

- 删除原因
- 可能的用途

## 读者 Context 检查

### 核心问题

> 「我的读者是谁？这个人在看这篇文章时会有哪些不明所以的地方？」

### 检查清单

| 检查项       | 问题表现                          | 改进方式             |
| ------------ | --------------------------------- | -------------------- |
| **背景假设** | 直接使用专有名词、缩写、梗        | 补充简短解释或脚注   |
| **自说自话** | 「就是那种...」但没说清楚是哪种   | 具体化描述           |
| **跳跃逻辑** | 从 A 直接到 C，缺少 B 的铺垫      | 补充过渡说明         |
| **隐含情绪** | 「实在是太...」但读者不知道为什么 | 先铺垫原因再表达情绪 |
| **信息差**   | 假设读者知道某件事的来龙去脉      | 简要交代背景         |

### 目标读者画像

根据博客定位，主要读者是：

- 十年后的自己
- 有 ACG 背景的「赛博邻居」
- 陷入类似困境、愿意深入对话的人

**不是**：不看动画的人、只想快速结论的人

### 检查流程

1. **标记可疑点** - 用 `<!-- TODO: 读者可能不懂XXX -->` 标记
2. **逐个讨论** - 这个点需要解释吗？怎么解释？
3. **确认后修改** - 补充必要的背景信息

## 大纲平衡检查

### 检查项

- **篇幅失衡**：某部分过长/过短
- **重点偏移**：次要内容占比过大
- **虎头蛇尾**：开头详细、结尾草率

### 输出建议

- 哪些部分需要扩展
- 哪些部分可以精简
- 哪些部分建议独立成文

## Review 节奏

1. **按 Markdown 标题为节点** - 每次 review 1-2 个标题下的内容
2. **每段 review 后等待确认** - 不连续处理多个段落
3. **启发式提问** - 对草稿或未完成内容，用提问启发，不替用户决定方向
