# 日记追记

快速追记一段内容到今天的日记。适合 agent session 结束时记录经验、随手记 TIL、
或任何不想走完整日记流程的场景。

## 前置条件

| 工具 | 类型 | 必需 | 安装方式 |
|------|------|----------|---------|
| 无 | — | — | 无外部依赖 |

## 使用时机

- Agent session 结束后，想把经验/教训记下来
- 随手记一个 TIL 或想法
- 工作中遇到值得记录的事，但不想启动完整日记流程

**不适用：**

- 完整日记写作 → 使用 `diary-assistant`
- 周报写作 → 使用 `weekly-report`

## 流程

```text
1. 定位日记文件        (~0min，自动)
2. 确定目标章节        (~0min，自动)
3. 追加内容            (~1min)
4. 可选：生成 Anki     (~0min，仅询问)
```

## 步骤 1：定位日记文件和写作偏好

日记配置与 `diary-assistant` 共用。

先读取 `~/.config/nini-skill/diary/user-config.md`。如果不存在，再读取
`diary-assistant` 的 `references/user-config.md`。两处都不存在时，询问日记目录。

找到配置后，如果存在 `## 写作风格` 章节，必须同时读取。将其中的格式、标点和语气
偏好叠加到步骤 3 的通用规则上。没有该章节时，仅使用通用规则。

**文件名**：`{YYYY-MM-DD}.md`，日期取当天。

文件不存在时，使用基础标题创建：

```markdown
# {YYYY-MM-DD}
```

## 步骤 2：确定目标章节

读取现有日记，确定内容追加位置：

| 内容类型 | 目标章节 | 缺少时创建 |
|-------------|----------------|-------------------|
| 工作经验或技术内容 | `## 2 Work Log` | 是 |
| 个人记录或想法 | `## 3 Record` | 是 |
| 用户指定章节 | 按指定位置 | 是 |

日记没有章节结构时，追加到文件末尾。

## 步骤 3：追加内容

将用户提供的内容或从当前对话整理出的内容写入目标章节。

### 通用规则

- **日记写给未来的自己，不解释已经知道的事。** 保留关键词、日期和上下文引用等线索，
  不展开完整故事。目标是「提醒」，不是「文章」。
- **记录人的思考，不记录 agent 的调试过程。** 日记应保留写作者的策略选择、明确表达的
  原则、思维变化和经验反思。pre-commit hook 损坏、MCP API 缺口、沙箱 SSL 问题、
  漏执行 `git add` 或 plugin 安装命令都属于 agent 的操作记录，应写入 agent memory
  （`feedback_*.md`）或工作流文档，不写入日记。整理会话时判断：「三个月后，写作者
  是否仍想回忆这件事，还是它只属于 agent 完成任务时遇到的障碍？」
- **避免旁白式或总结式开场。** 删除「今天最大的洞察」「之前一直以为……但其实」
  「借这个机会」「值得记录的是」等 AI 文章式表达。英文中的
  "The key insight today is" 和 "What I realized is" 同样删除。
- **正文优先使用连续段落，不在单条记录中堆叠小标题。** 小标题会诱导 AI 为保持结构
  平衡而补充无效内容。每条记录使用一个 H3 即可；存在多个逻辑单元时，使用
  `## 写作风格` 中配置的行内分隔符，不再增加标题。
- 使用项目符号或短段落。
- 保留用户原有语气，不要过度润色。
- 同一章节存在多条记录时添加时间前缀：
  `**14:30** — 记录内容`
- 用户提供原始对话上下文（例如 "record what we just did"）时，只整理关键结论。

### 写作规则优先级

日记包含中文正文时，按以下顺序应用规则。发生冲突时，上层规则覆盖下层规则：

1. **`tech-doc-style-chinese` skill 规则**（最高优先级）：使用 `「」`，不用 `“”`
   和感叹号；处理中英文留白；统一 `GitHub`、`Codeup`、`OpenSpec` 等术语大小写；
   避免「兜底」「赋能」「抓手」等词；应用错误翻译表和「克制、直接、可执行」的语气。
   当前会话未加载该 skill 时，仍应采用其核心规则。
2. **上述通用规则**：保留提醒线索、避免旁白、过滤 agent 操作过程、正文优先于小标题。
3. **用户配置中的个人风格**：读取 `## 写作风格` 中的加粗摘要、分隔符、引号、破折号
   和语气偏好，只应用不与前两层冲突的配置。

**常见冲突处理**（以 `tech-doc-style-chinese` 为准）：

- `register: 工程师口语` 与「克制、直接、可执行」冲突时，采用更克制的技术文档语气；
  `provider`、`callback`、`submodule` 等技术术语仍可保留英文。
- `self_deprecating_close: true` 应尽量弱化。确有必要的结尾应简短、基于事实，不使用
  `（笑）`、`（埋雷）` 等自嘲式装饰。
- `parenthetical_asides: true` 允许简短的括号内技术说明，但不得使用旁白式插话。
- `quote_style`、`em_dash`、`bold_as_tldr`、`section_divider` 通常与上层规则一致。

## 步骤 4：可选生成 Anki 卡片

内容符合 TIL（Today I Learned）模式时，询问：

```text
"Want to generate Anki cards from this? (invoke anki-card-generator)"
```

只询问，不自动调用。

## 常见问题

| 问题 | 处理方式 |
|-------|-----|
| 未找到日记路径 | 询问路径，并建议保存到 `~/.config/nini-skill/diary/user-config.md` |
| 文件编码异常 | Obsidian 使用 UTF-8，写入时保持 UTF-8 |
| 章节标题不匹配 | 先读取文件，再沿用现有章节名称 |
