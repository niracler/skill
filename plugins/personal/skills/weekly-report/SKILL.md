# 软件研发周报

软件研发周报生成助手。自动扫描 `~/code/` 下所有项目的 Git 日志，结合前几周周报
做对比，生成结构化的三段式周报，写入 Obsidian 日记。

## 前置条件

| 工具 | 类型 | 必需 | 安装方式 |
|------|------|----------|---------|
| Git | CLI | 是 | `brew install git` |
| reminders-cli | CLI | 是 | `brew install keith/formulae/reminders-cli` |
| macOS Calendar | 系统 | 是 | 系统内置，通过 `osascript` 访问 |
| gh | CLI | 否 | `brew install gh` 后运行 `gh auth login`，用于获取 GitHub PR 和 Issue |

### 路径依赖

| 路径 | 用途 | 必需 |
|------|---------|----------|
| `~/code/*/` | 项目仓库 | 是 |
| `~/code/*/repos/*/` | 嵌套仓库 | 否，存在时自动扫描 |
| Obsidian 日记目录 | 包含 Work Log 的日记文件和周报输出位置 | 是 |

扫描 `~/Library/Mobile Documents/iCloud~md~obsidian/` 自动识别 Obsidian vault。
未找到时询问用户。

> 加载 skill 时不要主动检查工具和路径。命令因缺少工具而失败时，跳过对应数据源，
> 使用其余可用数据继续。

## 工作流概览

流程顺序：**确认周期** → **查找历史周报** → **并行收集数据** → **生成草稿** →
**审阅并写入** → **安排下周计划**。

```text
┌──────────────────────────────────────────────────────────────────┐
│                weekly-report 完整流程（~20min）                    │
└──────────────────────────────────────────────────────────────────┘

  ┌──────────────┐
  │ 用户触发周报  │
  └──────┬───────┘
         │
         ▼
  ┌────────────────────────────────────┐
  │ 1. 确认周期 + 定位文件    (~1min)   │
  └──────┬─────────────────────────────┘
         │
         ▼
  ┌────────────────────────────────────┐
  │ 2. 查找前几周周报         (~1min)   │
  │    提取上周「下周工作计划」         │
  └──────┬─────────────────────────────┘
         │
         ▼
  ┌─────────────────────────────────────────────────┐
  │ 3. 并行数据收集（subagent）          (~3-5min)   │
  │    ┌─ 本周日记 Work Log（各工作日）              │
  │    ├─ Git log（~/code/*/ 全扫描）                │
  │    └─ GitHub PR/Issue（gh CLI）                  │
  └──────┬──────────────────────────────────────────┘
         │
         ▼
  ┌────────────────────────────────────┐
  │ 4. 交叉比对 + 生成草稿    (~3min)   │
  │    上周计划 vs 本周实际             │
  └──────┬─────────────────────────────┘
         │
         ▼
  ┌────────────────────────────────────┐
  │ 5. 用户审阅 + 修改        (~5min)   │
  │    展示草稿 → 用户调整 → 写入日记  │
  └──────┬─────────────────────────────┘
         │
         ▼
  ┌────────────────────────────────────┐
  │ 6. 创建下周日程 + 提醒    (~3min)   │
  │    Calendar 时间块                 │
  │    Reminders 任务列表              │
  └──────┬─────────────────────────────┘
         │
         ▼
      ┌──────┐
      │ 完成  │
      └──────┘
```

## 步骤 1：确认周期和目标文件

- **报告周期**：本周一至周五。
- **目标文件**：当天的日记，通常是本周最后一个工作日。
- **写入位置**：`## 2 Work Log` 下的 `### 软件研发周报（M/D - M/D）`。
- **日记路径**：在 `~/Library/Mobile Documents/iCloud~md~obsidian/` 中扫描
  Obsidian vault，未找到时询问用户。

当天是周六或周日时，默认使用上周五的日记。继续前需要用户确认。

## 步骤 2：查找历史周报

扫描最近 14 天的日记，查找标题 `### 软件研发周报`。

从最近一篇历史周报中提取：

1. **「下周工作计划」章节**：作为计划与实际完成情况的比较基准。
2. **格式和风格**：沿用项目分组、细节层级和加粗方式。

没有历史周报时跳过比较并继续；首次生成周报时属于正常情况。

## 步骤 3：并行收集数据

并行启动 subagent。三个数据源相互独立，应尽量同时执行。某个数据源失败或不可用时，
跳过并继续。

### 3a：日记 Work Log

读取报告周期内每个工作日的日记，提取每天 `## 2 Work Log` 下的内容。

这是信息最丰富的数据源，通常包含 Git commit 和 PR 标题无法表达的背景、决策与细节。
写周报时优先使用这些内容。

### 3b：收集 Git 日志

扫描 `~/code/*/` 和 `~/code/*/repos/*/` 下的 Git 仓库，并分别运行：

```bash
git log --oneline --since="{monday}" --until="{saturday}" --author="{user}" --all
```

`{user}` 取自 `git config user.name`。

按项目和仓库分组 commit，用于补充日记中遗漏的工作。

### 3c：GitHub 活动（`gh` 可用时）

```bash
# PRs authored this week
gh search prs --author=@me --created="{monday}..{saturday}" --json repository,title,number,state,url

# Issues authored/assigned this week
gh search issues --author=@me --created="{monday}..{saturday}" --json repository,title,number,state,url
```

收集已合并 PR、未关闭 PR 和已关闭 Issue。PR 编号和标题作为具体参考信息。

### 数据源优先级

同一项工作出现在多个数据源时合并处理：

| 优先级 | 数据源 | 优势 |
|---|---|---|
| 1 | 日记 | 上下文最完整，包含用户原话、决策和理由 |
| 2 | GitHub | 提供 PR 和 Issue 等具体引用 |
| 3 | Git 日志 | 发现未记录工作并核对事实 |

日记提供叙述，GitHub 提供可追溯信息，Git 日志补充遗漏。

### 数据源降级

| 不可用的数据源 | 处理方式 |
|---|---|
| 部分日期缺少日记 | 使用已有日期，并标记缺口 |
| 未安装 `gh` 或认证失败 | 跳过 GitHub，并标记「GitHub 数据未获取」 |
| Git 日志为空 | 项目可能只有会议或设计等非代码工作，检查日记 |
| 所有远程数据源失败 | 仅使用日记和 Git 日志继续 |

## 步骤 4：交叉核对并生成草稿

### 计划完成情况检查（存在历史周报时）

将上周「下周工作计划」与本周实际产出比较：

| 上周计划 | 本周状态 | 处理方式 |
|---|---|---|
| 已完成 | 完成 | 写入本周工作总结 |
| 部分完成 | 进行中 | 在本周总结中说明进度，并移入下周计划 |
| 未开始 | 遗漏 | 在其他事项中标记，已知原因时一并说明 |
| 完成计划外工作 | 新增 | 作为新项目写入本周总结 |

### 项目分组

不要硬编码分组。按以下信息决定：

1. **历史周报**：存在时沿用项目显示名称和分组。
2. **日记**：使用作者已经采用的项目名称。
3. **仓库结构**：适当把嵌套仓库归入同一项目。

省略本周没有活动的项目。

### 生成草稿

按 [report-template.md](references/report-template.md) 生成三个章节。

**本周工作总结:**

- 按项目分组，优先沿用已有显示名称。
- 每个项目最多 1～2 条，压缩为核心交付结果。
- 每条采用 `**加粗关键词**：` 加业务语言说明。
- 使用接口数、页面数和参数个数等领域数字，不使用 commit 数或 PR 编号等内部数字。
- 完全省略 CI/CD 变更、版本升级、SDK 子版本发布、上游开源贡献，以及 CLAUDE.md、
  openspec schema 等内部流程改进。
- sunlite 和 sylsmart 是独立项目，不得合并。

**下周工作计划:**

- 来源包括顺延事项、日记、Reminders 和用户明确输入。
- 已知时间估算时写入，例如 `（2.5 天）`。
- 每条采用 `**加粗关键词**：` 加业务语言描述的预期结果。
- sunlite 和 sylsmart 必须使用独立标题和时间分配。

**其他事项:**

- 写入上周未完成的顺延事项，并简要说明原因。
- 写入交接、阻塞和跨团队配合事项。
- 保持简洁，不在括号中补充技术解释。

## 步骤 5：用户审阅并写入

1. 展示完整草稿。
2. 等待用户反馈。
3. 应用修改。用户未提到的条目视为认可，不得删除。
4. 将最终版本写入日记的 `## 2 Work Log`。

## 步骤 6：安排下周计划

周报写入后，解析「下周工作计划」，创建下周 Calendar 日程和 Reminders，使计划成为
可执行的日程事项。

### 解析计划事项

从「下周工作计划」的每条内容中提取：

- **任务名称**：加粗关键词。
- **项目**：所属项目标题。
- **时间估算**：存在 `（N 天）` 时读取。

### 创建提醒事项

先使用 `reminders show-lists` 检查可用列表，再写入合适的列表。优先使用「提醒」，
否则使用第一个与工作相关的列表：

```bash
reminders add "提醒" "{project}: {task name}" --due-date "next monday"
```

任务指定日期时，例如「周三前完成」，使用该日期。

### 创建 Calendar 时间块（可选）

项目包含时间估算时，建议创建 Calendar 时间块。使用下周的实际日期，不使用相对偏移：

```bash
# Example: create a time block for next Monday (2026-03-16) 9:00-17:00
osascript -e '
tell application "Calendar"
    tell calendar "工作"
        set startDate to date "2026年3月16日 09:00:00"
        set endDate to date "2026年3月16日 17:00:00"
        make new event with properties {summary:"sylsmart: auth 接口适配", start date:startDate, end date:endDate}
    end tell
end tell'
```

### 默认值

- **Calendar 名称**：`Work` 后跟一个半角空格。使用
  `osascript -e 'tell application "Calendar" to get name of calendars'` 核对。
- **时间块**：9:30～11:45，作为上午专注时段，不占用全天。
- **不创建全天日程**：始终设置具体开始和结束时间。
- **Reminders 列表**：`提醒`。

### 日期计算

使用 `current date` 加天数偏移计算下周日期。不要硬编码日期字符串，避免 macOS 因
区域设置产生解析错误。

```applescript
set baseDate to current date
set time of baseDate to 0
set nextMon to baseDate + ({offset} * days)
set startTime to nextMon + (9 * hours) + (30 * minutes)
set endTime to nextMon + (11 * hours) + (45 * minutes)
```

先展示日程建议，得到用户确认后再创建：

```text
下周日程建议（Work 日历，9:30-11:45）：
  周一 3/23: sylsmart - xxx（1 天）
  周二-周三 3/24-3/25: sunlite - xxx（2 天）
  周四 3/26: HA 集成 - xxx（1 天）
  周五 3/27: buffer / 周报

确认创建到 Calendar 和 Reminders 吗？
```

## 重要行为规则

**不得删除用户未提到的条目。** 审阅时，用户通常只指出需要修改的部分；没有反馈表示
认可。不要为了加入修改内容而删除原本正确的条目。

**使用业务语言，不堆叠技术术语。** 主管不阅读代码。说明完成了什么，以及它带来什么
结果，使用接口、页面、配置参数和用户能力等领域词汇。不得写入 Router Factory、
Playwright E2E、migration downgrade 等实现细节，CLAUDE.md、openspec review schema
等内部流程名称，版本升级、commit 或 PR 数量，以及 CI/CD 细节。这些信息可用于收集，
但最终周报必须转换为业务结果。

**使用具体的业务数字。** 可以写 14 个接口、17 个页面、4 个配置参数、18 个待确认问题；
不要写 13 个 commit、PR#78 或 91% coverage。

**沿用历史周报风格。** 存在历史周报时，保持语气、细节层级和格式选择一致，使连续多周
形成统一叙述。

## 常见问题

| 问题 | 处理方式 |
|-------|-----|
| 未找到日记 | 核对 Obsidian vault 路径和 iCloud 同步状态 |
| 未找到历史周报 | 首次生成时属于正常情况，跳过计划完成情况检查 |
| `gh search` 返回 0 条结果 | 检查日期范围格式，并运行 `gh auth status` 核对认证状态 |
| 未找到 Calendar「工作」 | 列出可用日历并使用现有名称 |
| 未找到 Reminders 列表 | 运行 `reminders show-lists` 并使用可用列表 |
| Git 日志包含上游 commit | 使用 `--author` 过滤上游内容 |
